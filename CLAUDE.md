# re-hana-counter

モバイル向けカウンターWebアプリ。
**仕様は `test/` 配下のテストコードを正とする**（説明は日本語）。
要件文書はユーザーが必要に応じてプロンプトで渡す。テストと矛盾したらテストに従い、
疑問があればユーザーに確認する。

## コマンド

- `bun test` — テスト実行
- `bun run dev` — 開発サーバ (Vite)
- `bun run build` — 型チェック + 本番ビルド
- `bun run typecheck` — 型チェックのみ（アプリ・Worker・テストの全プログラム）
- `bun run lint` / `bun run format` / `bun run knip` — 品質チェック

## 実装方針

### TDD — 実装より先にテストを書く（必須）

- **いかなる実装コードも、それを検証するテストより先に書いてはならない。**
- 手順は必ず Red → Green → Refactor の順で進める:
  1. `test/` の該当する `test.todo()` を実テスト（アサーション付き）に起こす
  2. `bun test` を実行し、そのテストが**失敗することを確認する**（Red）
  3. テストを通す最小限の実装を書く（Green）
  4. テストが通る状態を保ったままリファクタリングする（Refactor）
- テストのない振る舞いを実装に追加しない。仕様変更はまずテストに反映する。
- **テストファイルはそれ自身が仕様書として機能するよう書く。**
  describe / test の名前と構造だけで仕様が読み取れるよう日本語で記述する。
- テスト名（describe / test）に主観的・曖昧な表現を使わない。
  条件と期待結果を判定可能な形で書き、可能な限り具体値を併記する
  （「正しく」「適切に」「即座に」「一括で」等の副詞、「左」「右端」等の位置語は使わない）。
- テスト名は、そのテストのアサーションが実際に検証する内容と一致させる
  （検証していない性質をテスト名で主張しない）。
- テスト名・テストファイルの説明コメントにコード片（関数呼び出し・演算子・
  シグネチャ）を書かない。具体例は入力値と期待値を日本語の文で表現する
  （関数名・属性名・パスなど、検証対象を指す識別子と値はそのまま書いてよい）。
- テストが検証するのは**振る舞いのみ**。色・寸法・配置などのデザインはテストしない
  （デザインは CSS と目視確認で担保する）。

### 最小実装・性能

- テストを通す最小限のコードのみ実装する。抽象化（共通化・汎用化）は、
  同じ処理が2箇所目で必要になった時点で導入する。
- 機能追加を伴わない変更でバンドルサイズを増やさない。
  画像ファイルは配信物に追加しない（favicon 用の `public/hana.svg` を除く）。
  マークは CSS（`clip-path` / `radial-gradient`）、装飾アイコンはインライン SVG で描く。
- state はそれを使用する最下層のコンポーネントに配置する。
  `memo` / `useMemo` / `useCallback` は、計測で性能問題を確認した箇所にのみ使う。
- **アニメーションは使用しない。**
  CSS の `transition` / `animation` プロパティ、擬似クラスによる押下時の
  スタイル変化を実装しない。例外はユーザーの事前承認を必須とする。

### 依存パッケージを増やさない

- ランタイム依存は **react / react-dom / hono / react-router のみ**。
  UI コンポーネントライブラリ、CSS フレームワーク、アイコン集、状態管理、
  ユーティリティ（lodash 等）、日付・フォームライブラリは追加しない。すべて自前実装する。
- React Router で使用する API は BrowserRouter / Routes / Route / Link に限定する。
  loader / action 等のデータ API は使用しない。
- スタイルは素の CSS。色・寸法は CSS 変数で一元管理する。
  CSS-in-JS やプリプロセッサは使わない。
- 状態管理は `useState` / `useReducer` と props で完結させる。
  React Context は使用しない。導入はユーザーの事前承認を必須とする。
- 依存パッケージの追加はユーザーの事前承認を必須とする。

### コード構成

- **実装コード（`src/` / `worker/`）にはコメントを書かない。**
  意図の補足は、コメントではなく命名と関数分割で表現する。
  （テストファイルは仕様書を兼ねるため、この限りではない）
- 増減・下げ止まり・リセット・バリデーション・永続化のロジックは
  UI から分離した純粋関数として `src/lib/` に置く。
- localStorage アクセスは Storage を引数注入する形にする
  （`loadState(storage)` / `saveState(state, storage)` — test/storage.test.ts 参照）。
- カウンター構成の追加・削除・変更が `src/lib/state.ts` の `sections` 配列の
  変更のみで完結するようにする。

## コーディング規約

- インデントは半角スペース2つ（Prettier / .editorconfig 管理）。
- コミット前に `bun test` / `bun run typecheck` / `bun run lint` /
  `bun run format:check` を通す。
